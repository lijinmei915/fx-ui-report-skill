# R08 务实表达规则

## 目标

防止简报变得抽象、虚浮或行业通用化。本规则适用于 R03 痛点叙事、R04 竞品话术和 R06 最终行动建议。

## 核心原则

在确认客户真正要解决的问题之前，不要急着告诉客户产品能做什么。

使用：

```markdown
我先确认一下，我理解的场景是不是这样：{concrete_scene}
```

不要使用：

```markdown
我们可以赋能贵司数字化转型。
```

## 禁止作为主切入点的词

以下词不能作为会议主切入点：

- 赋能
- 数字化转型
- 智慧化
- 生态
- 第二增长曲线
- 全链路升级
- 没有具体前后指标的“提升效率”

只有客户自己先使用这些词时，简报才可以引用；即便引用，也必须翻译成具体业务场景或验证问题。

## 具体场景规则

每个首选痛点或 P0 行动，至少包含以下一项：

- 客户原话。
- 具体业务流程场景。
- 可衡量的前后变化目标。
- 可在会议现场提问验证的问题。

如果都没有，必须标为假设。

## 5 分钟行动测试

R06 中的每个行动，都必须能在会议中 5 分钟内完成。

错误示例：

```markdown
P0：强化客户对服务价值的认知。
```

正确示例：

```markdown
P0：问客户："现在服务超时最多发生在哪一步，派单、到场，还是完工确认？"
```

## 推断格式

对没有完全证据支撑的判断，正文只保留轻量推断标记和验证问题，避免把来源说明塞进主简报：

```markdown
推断：{claim}（置信度：推断/猜测）
验证问题：{one question}
```

证据类型和数据来源统一放在 R06 简报末尾的 `📎 参考来源` 弱引用区。

不要把推断写成客户已经确认的痛点。

## 竞品表达

不要说“我们更好”。使用场景适配语言：

```markdown
如果你们现在最难的是 {customer_scene}，我们在 {specific_capability} 上更合适。
如果你们更看重 {competitor_strength}，竞品可能更省心。
```

这样更可信，也能避免客户产生防御心理。

## 最终检查

输出 R06 前，检查是否存在：

- 没有具体场景支撑的抽象词。
- 没有问题或可观察下一步的行动。
- 没有客户场景的竞品对比。
- 把行业假设写成客户事实。
- 在第一个行动前铺垫超过一段背景。
- 在主体简报中逐条堆叠 `来源：...` 或 `依据：...`。

发现后先修正，再输出简报。
