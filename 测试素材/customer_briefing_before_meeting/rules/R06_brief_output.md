# R06 见客简报输出

## 目标

把 R01-R05 的分析组合成最终见客前简报。本规则不引入新事实，只负责筛选、压缩和转化为会前结论与会议动作。

## 触发条件

R05 完成后始终执行。

## 输入依赖

- R01：客户/商机基础信息和即时风险。
- R02：决策链和角色缺口。
- R03：首选痛点、验证问题、必选条件。
- R04：竞争态势和场景化竞品话术。
- R05：见客就绪度和最大缺口。
- Memory：只使用已经读取过的客户事实。

## 输出长度

- `lite`：8-12 行。
- `normal`：核心主简报一屏内，目标 650 个汉字以内；末尾弱引用不计入主体。
- `full`：先输出 normal 简报，再追加展开内容。

第一段主简报必须保证 AE 一分钟内读完。

## 视觉扫读规则

- 默认使用少量 emoji，优先放在模块标题和关键状态上，帮助 AE 在移动端快速定位。
- 推荐模块 emoji：`🧭 会前结论`、`🪪 身份标签`、`🎯 决策锚`、`⚠️ 预警`、`📊 就绪度`、`👥 人力地图`、`📅 博弈时间线`、`✅ 今日行动`、`📎 参考来源`。
- 不要在每个字段、每个风险或每条行动前都加 emoji；正文以销售动作和风险判断为主。
- 风险等级可使用 `🔴/🟡/🟢`，但必须服务于严重度判断，不作为装饰。

## 会前结论规则

- 会前结论必须放在简报最顶部，先给现场打法，再展开证据。
- 一句话判断必须同时说明：今天主目标、最大风险、关键打法。
- 表格只保留 5 项：今天主目标、最大风险、最关键的人、第一句话/第一个问题、绝对不要做。
- “第一句话/第一个问题”必须是 AE 可以直接照着问的现场语言。
- “绝对不要做”必须是明确禁忌动作，不写成泛泛提醒。
- 会前结论不新增事实，只从 R01-R05 中抽取最关键判断。
- 正文不逐条展示 `来源：...` 或 `依据：...`；证据口径统一放到末尾弱引用区。

## HTML 结构与防重叠规则

生成 HTML / 移动端版本时，内容不变，但必须按“信息优先级 + 自适应容器”安排结构：

- 首屏顺序固定为：顶部客户身份栏 → 会前结论 → 致命预警 → 就绪度 → 今日行动看板。
- 不使用固定高度承载动态内容；顶部栏、会前结论、预警横幅都必须由内容自然撑开。
- 不使用绝对定位叠放正文、就绪度、主题切换或徽章；主题切换在桌面端放入顶部栏右侧，在移动端独立固定。
- 顶部客户身份栏只放身份、阶段、金额、赢率、行业规模、预算/Coach/致命风险、就绪度；长字段允许换行，不能压缩到覆盖下一行。
- 会前结论 5 项在桌面端使用自适应卡片网格，最小列宽不低于 220px；中等宽度可自动换成 2-3 列，不能强制 5 列。
- 桌面端主体优先使用四块行动看板：今日目标、3 件预警、人力地图、博弈时间线；窗口不足时自动降为两列。
- 移动端只保留最关键的会前结论和预警，人力地图、时间线、基础信息、痛点放入展开区。
- 所有长句、客户名称、风险描述和问题话术必须允许换行，避免 `nowrap` 用在正文内容上。
- HTML 生成后检查首屏：顶部栏、会前结论、预警横幅、就绪度之间不能互相遮挡，按钮和徽章不能覆盖正文。

## 数据来源展示规则

- 主体简报优先表达结论、风险和现场动作，不让来源字段打断阅读。
- 对没有直接证据支撑的判断，只在正文保留 `推断` 或 `猜测` 标记，并给出现场验证问题。
- 数据来源统一放在简报最后，用低强调弱引用格式展示。
- Markdown 默认使用一行 blockquote；如果输出平台支持 `<details>`，可以折叠展示。

默认格式：

```markdown
> 📎 参考来源：CRM 客户/商机/联系人/跟进记录、客户 Memory、必要公开信息及行业/竞品参考。正文中的“推断/猜测”仅用于会前准备，需在现场验证。
```

可折叠格式：

```markdown
<details>
<summary>📎 参考来源</summary>

- CRM：客户、商机、联系人、最近跟进记录。
- Memory：历史承诺、决策链变化、竞品阶段、预算/审批风险。
- References：行业基准、竞品参考、CRM 字段说明。
- 推断项需在现场验证，不视为客户确认事实。

</details>
```

## 人力地图规则

- `normal` 默认输出轻量人力地图，放在见客就绪度之后、今日行动之前。
- 只保留 3-5 个会影响本次会议的人：今天参会人、EB/EB 候选、Coach、Gatekeeper、TB、关键 UX。
- 每行必须包含“今天打法”，不要只写职位或关系。
- 不明确时写“未知/待确认”，并把识别问题写进今天打法。
- 如果没有任何人物信息，保留 1 行“关键决策人未知”，今天打法设为识别 EB。

## 博弈时间线触发规则

默认不强制输出博弈时间线。满足任一条件时输出，放在人力地图之后、今日行动之前：

- 老客户、续约、增购、复购或历史合作较长。
- 已经多轮跟进，存在承诺未兑现、会议反复、阶段停滞。
- 竞品介入、预算/审批变化、关键人变化。
- 用户明确要求“时间线、历史脉络、博弈过程、项目历程”。

博弈时间线在 `normal` 中最多 3 个节点；`full` 可扩展到 5 个节点。每个节点必须写“对今天的影响”或“今天要验证的问题”。不要写完整大事记。

## 字段选择规则

### 身份标签

只选择能帮助 AE 决定今天怎么行动的标签：

- 客户类型：战略型、成长型、稳健型、收缩型、未知。
- 情绪温度：积极、中性、冷淡、负面、未知。
- 当前阶段：初次接触、需求确认、POC、方案、商务谈判、续约、重启。

不确定时写“未知”，并说明关键缺口；来源类型收口到末尾弱引用。

### 会议目标

只选择一个主目标：

- `签约`：见客就绪度 >= 80，EB 已知，预算已确认，无致命风险。
- `高层承诺`：需要 EB 参与或明确表态。
- `探测预算`：预算状态未知或审批链路不清。
- `排除障碍`：Gatekeeper、必选条件、竞品或采购风险活跃。
- `重建上下文`：近期互动过旧或 CRM 历史很薄。

不要选择比证据更激进的目标。

### 底线

底线必须是约束，不是口号：

- 不承诺无法支撑的必选功能。
- 价值和决策链未确认前，不主动让价。
- 非 EB 会议不进入最终商务谈判。
- 不直接攻击竞品。

### 预警

按严重度和与今日参会人的相关性选择前三个风险。风险证据优先级：

1. R01 中的承诺未兑现、预算未知、结单压力、商机停滞。
2. R02 中的 EB 未知、Coach 薄弱、Gatekeeper、角色错配。
3. R03 中的必选条件无法满足或痛点证据弱。
4. R04 中的 C3/C4 竞品或差异化弱。
5. 客户 memory 中的历史风险模式。

每条预警必须有现场应对动作。真实风险不足三条时，可以少于三条，不要凑数。

### 行动建议

每个行动都必须通过这个测试：

```text
AE 能否在会议中 5 分钟内完成或问出来？
```

错误示例：

```markdown
P0：深化客户关系。
```

正确示例：

```markdown
P0：问林总："这次系统上线后，您最想先看到哪个指标变化？"
```

## Full 模式

normal 简报之后，只追加用户要求的部分：

- R02 完整决策链、人力地图和博弈时间线详情。
- R03 痛点矩阵。
- R04 竞争态势详情。
- R09 历史财务/合作详情。
- 使用 `assets/templates/visit.html` 和 `assets/styles/briefing.css` 生成 HTML/移动端版本。

normal 输出不要包含归档的 out-of-scope 素材。
