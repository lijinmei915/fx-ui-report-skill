# 见客前客户简报

## 目标

为 AE 在见客户前生成一份短、准、能指导行动的客户简报。简报必须帮助 AE 回答：

- 今天这场会怎么打，最先抓什么？
- 今天这场会的主要目标是什么？
- 当前见客就绪度是多少，最大信息缺口是什么？
- 今天必须守住哪些风险？
- 今天要影响谁、绕开谁、借力谁？
- 现场应该问哪句话、做哪个动作？

默认输出 Markdown 简报。只有用户要求移动端、可分享或 HTML 版本时，才使用 HTML 模板。

## 适用场景

使用本技能处理：

- 单客户见客前准备。
- CRM 触发的“即将拜访”或“准备出发”场景。
- `/briefing [客户名]`、`/briefing [客户名] --style=lite|normal|full`。
- 用户要求见客就绪度、决策链、人力地图、会议底线、风险预警或今日行动建议。

不要用于：

- 团队交接或 AE 工作移交。
- 高层陪访、领导汇报或升级汇报。
- 客户组合盘点、团队周报、销售经理 review。
- 完整首访/复访策略、调研会问题体系或访谈节奏设计；交给 `first-visit-strategy`。
- 深度需求穿透、痛点还原或预算三层穿透；交给 `needs-discovery`。
- 完整决策链诊断、EB/Sponsor识别或隐性权力中心判断；交给 `decision-chain`。
- Coach识别、分级和培养；交给 `coach-development`。
- 单商机竞品攻防；交给 `competition-intel`。
- 会后记忆更新作为主任务，除非用户明确要求在简报后更新 memory。

本技能负责生成见客前行动简报，可以汇总客户事实、风险、决策链、竞品、需求和商机信息，但不做深度业务诊断。用户要完整首访策略转 `first-visit-strategy`，要需求诊断转 `needs-discovery`，要决策链诊断转 `decision-chain`，要Coach经营转 `coach-development`，要竞品攻防转 `competition-intel`。

## 需要获取的信息

必需：

- 客户名称或 CRM 客户 ID。
- 会议上下文：今天见谁、会议目的、当前商机阶段。

优先获取：

- CRM 中的客户、商机、联系人、最近跟进记录。
- 用户明确要求外部动态时，再补充客户近期公开信息。
- `memory/customers/[客户名称]/` 下的客户长期记忆。

如果无法识别客户，或 CRM 查询到多个候选客户，先让用户确认，不要直接生成简报。

## 数据优先级

按以下顺序使用证据：

1. CRM 记录和用户提供的会议上下文。
2. 客户 memory 文件。
3. 公开信息，仅在可用且与当前会议相关时使用。
4. 行业基准和竞品参考。
5. 明确标注的推断。

不要用公开信息或推断覆盖 CRM 事实。所有推断必须标注置信度：`确认`、`推断` 或 `猜测`。

## 数据处理流程

1. 识别客户记录。
   - 需要从 CRM 查询时，先读取 `references/cli_reference.md`，按企业 CLI 规范查询 `AccountObj`。
   - 如果有多个候选客户，要求用户选择。

2. 拉取关联 CRM 数据。
   - 按 `references/cli_reference.md` 先查 normal 模式默认数据源；只有 full 模式或用户要求历史/财务/风险分析时，再查扩展数据源。
   - 客户：名称、`_id`、行业、规模、状态、来源、最后跟进时间。
   - 商机：阶段、金额、赢率、预计结单日期、状态、竞品、预算状态。
   - 跟进记录：最近 3-5 次互动、结论、情绪、承诺。
   - 联系人：姓名、职务、关键联系人标记、可能的 EB/TB/UX/Coach/Gatekeeper。

3. 读取客户记忆。
   - 如存在，读取 `memory/customers/[客户名称]/`。
   - 只有在会后沉淀、创建客户记忆或解释记忆结构时，才读取 `memory/template.md`。
   - 读取 memory 时只提取会影响本次见客的事实：决策链变化、痛点证据、承诺兑现、竞品阶段、预算/审批风险、必选条件。

4. 按输出深度加载规则。
   - `lite`：`rules/R01_customer_basics.md`、`rules/R05_ready_score.md`、`rules/R06_brief_output.md`、`rules/R08_pragmatic_style.md`。
   - `normal`：`rules/R01_customer_basics.md` 至 `rules/R06_brief_output.md`，再加 `rules/R08_pragmatic_style.md`。
   - `full`：normal 模式规则，再加 `rules/R09_history_financial.md` 和相关 references。

5. 计算见客就绪度。
   - 若数据可结构化，优先使用 `scripts/calc_ready_score.py`。
   - 若数据不完整，可定性计算，但必须列出导致扣分的缺失字段。

6. 生成简报。
   - 默认输出 Markdown。
   - 先输出“会前结论”，再展开身份标签、决策锚、预警、就绪度、人力地图和行动。
   - `normal` 默认输出轻量人力地图；仅在老客户、续约/增购、多轮跟进、竞品介入、历史承诺反复或用户要求时输出“博弈时间线”。
   - 使用少量 emoji 强化扫读，优先放在一级模块标题和关键状态上，不要给每行都加装饰。
   - 正文不逐条展示“来源/依据”，数据来源说明统一用弱引用格式放在简报最后。
   - HTML 资源：`assets/templates/visit.html` 和 `assets/styles/briefing.css`。

## 资源路由

- 需要从 CRM 拉取客户、商机、联系人、跟进记录、合同、订单、回款、CSM 预警、流失原因或竞品明细时，读取 `references/cli_reference.md`。
- 仅在需要把原始 CRM 输出或用户粘贴的 CRM 文本映射到 R01-R05 字段时，读取 `references/crm_schema.md`。
- 仅在需要行业基准、财务阈值、采购周期或采购时机判断时，读取 `references/industry_benchmarks.md`。
- 仅在已知、疑似或用户明确讨论竞品时，读取 `references/competitive_reference.md`。
- 仅在 `full` 模式或用户要求历史订单/回款分析时，读取 `rules/R09_history_financial.md`。
- 其他简报场景的归档素材已移至 `../shared_briefing_assets/out_of_scope_briefing_templates/`，正常见客简报不要加载。

## 输出契约

默认 `normal` 简报：

```markdown
## 见客前简报：{客户名称}

### 🧭 会前结论
**一句话判断：** 今天这场会应以 {签约/高层承诺/探测预算/排除障碍/重建上下文} 为主，最大风险是 {风险}。

| 项目 | 结论 |
| --- | --- |
| 今天主目标 | {只写 1 个} |
| 最大风险 | {只写 1 个} |
| 最关键的人 | {能改变局面的关键人} |
| 第一句话/第一个问题 | {AE 到场先怎么开口} |
| 绝对不要做 | {今天最该避免的动作} |

### 🪪 10秒身份标签
{客户类型} | {今日情绪温度} | {组织动机}
{一句话说明客户当前状态}

### 🎯 30秒决策锚
今日目标：{签约/高层承诺/探测预算/排除障碍}
底线：{不能承诺或不能让步的事项}
如果今天只能做一件事：{最关键动作}

### ⚠️ 20秒预警
1. {风险} | 应对：{现场动作}
2. {风险} | 应对：{现场动作}
3. {风险} | 应对：{现场动作}

### 📊 见客就绪度
信息完整度：{score}/100
痛点匹配度：{score}/100
竞争风险度：{score}/100
综合就绪度：{score}/100 {等级}
最大缺口：{缺失项}

### 👥 人力地图
| 人物 | 角色 | 当前立场 | 今天打法 |
| --- | --- | --- | --- |
| {姓名/未知} | EB/Coach/TB/UX/Gatekeeper | 支持/中立/摇摆/阻碍/未知 | {一句现场动作} |
| {姓名/未知} | {角色} | {立场} | {动作} |

### 📅 博弈时间线（条件触发）
- {时间}：{关键事件/承诺/竞品动作} → {对今天的影响}
- 今天：{本次会议在推进中的位置} → {必须验证的问题}

### ✅ 今日行动
P0：{一句可执行动作，含具体问题}
P1：{一句可执行动作}
P2：{一句可执行动作}
需要规避：{不能说/不能承诺/不能推进的事项}

> 📎 参考来源：CRM 客户/商机/联系人/跟进记录、客户 Memory、必要公开信息及行业/竞品参考。正文中的“推断/猜测”仅用于会前准备，需在现场验证。
```

`lite` 模式：

- 只保留会前结论、身份标签、见客就绪度、一个最高风险、一个最关键人、一个 P0 行动。

`full` 模式：

- 在主简报之后补充完整决策链、人力地图、博弈时间线、痛点矩阵、竞品分析和历史财务信号。
- 第一段主简报仍必须简短，确保 AE 一分钟内能读完。

## 务实表达规则

- 用具体会议语言，不用抽象战略话术。
- 不把“赋能”“数字化转型”“智慧化”“生态”作为主切入点。
- 每条行动建议都要有 5 分钟内能验证的标准。
- 痛点叙事必须包含具体场景，或给出可现场验证的问题。
- 竞品定位只能说“在这个场景下我们更合适”，不要说“我们更好”。

当结论没有直接证据支撑时，在正文中使用轻量推断标记，不展开逐条来源：

```markdown
推断：{claim}（置信度：推断/猜测）
验证问题：{AE今天可以问客户的一句话}
```

证据类型统一收口到简报末尾的“参考来源”弱引用，不要打断主简报阅读。

## 失败处理

- CRM 没有客户记录时，请用户确认客户名称或补充 CRM 数据。
- 没有近期跟进记录时，降低信息完整度，并把 P0 行动设为“重建上下文”。
- EB 不明确时，不要假设预算拍板人；把识别 EB 作为今日目标。
- 痛点只有行业假设支撑时，标注为假设，并给出验证问题。
- 见客就绪度低于 40 时，建议缩小会议目标或延期；除非用户明确说明会议不能改期。

## 会后记忆更新

用户要求“更新记忆”“记录本次拜访”“沉淀到 memory”时，读取 `memory/template.md`，只记录会影响下次见客的信息：

- 决策链：EB、Coach、Gatekeeper、TB、UX 的新增或变化。
- 痛点：客户原话、证据等级、产品匹配度、下次验证问题。
- 承诺：我方或客户承诺、截止日期、兑现状态。
- 风险：预算、审批、竞品、必选条件、交付或回款风险。
- 下次提醒：下次见客前必须守住的 1-3 件事。

不要把完整会议纪要塞进 memory；CRM 已有事实不重复写，除非需要标注“下次见客必须提醒”。
