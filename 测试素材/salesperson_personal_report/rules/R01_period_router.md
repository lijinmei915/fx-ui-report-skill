# R01 - 报告周期路由

## 目标

先判断用户要的是个人日报、个人周报还是个人月报，再决定读取哪些规则和输出什么粒度。

本规则始终先执行。

## 周期判断

| 用户表达 | 报告类型 | 读取规则 |
|----------|----------|----------|
| 今天、昨日、昨天、某天、某一日、具体日期、日报、每日总结 | 个人日报 | R08、R02、R05、R06、R07、R09；用户要求 HTML 时加 R10 |
| 本周、上周、某一周、某个日期所在周、Wxx、YYYY-WXX、周报、周总结 | 个人周报 | R08、R03、R05、R06、R07、R09；用户要求 HTML 时加 R10 |
| 本月、上月、某一月、具体月份、月报、月度复盘 | 个人月报 | R08、R04、R05、R06、R07、R09；用户要求 HTML 时加 R10 |

如果用户没有说明周期，优先追问一句：

你想看个人日报、周报还是月报？如果不指定，我默认生成个人周报。

如果用户同时提到多个周期，例如“本周和本月”，按用户顺序分别输出，不混成一个报告。

## 输出格式判断

识别用户是否要求 HTML 输出。以下表达表示 `output_format = html`：

- HTML
- html
- 网页版
- 手机查看
- 自适应页面
- H5
- 浏览器打开
- 做成页面

HTML 是输出格式，不是报告周期。用户说“生成 HTML 版个人周报”时：

- 报告周期 = 个人周报
- 输出格式 = HTML
- 仍执行个人周报完整分析流程
- 额外执行 `R10_html_report_output.md`
- 最后仍执行 CRM 日志保存确认

## 输出粒度判断

识别用户希望生成简洁版还是详细版，设置 `output_detail`：

| 用户表达 | output_detail | 处理 |
|----------|---------------|------|
| 简洁版、摘要版、快速看、给老板看、只要重点、三分钟看完 | concise | 输出简洁版 |
| 详细版、展开、完整、明细、逐个客户展开、复盘过程、全部跟进记录 | detailed | 输出详细版 |
| 直接生成、快点、先给我看、先出一版 | concise | 不追问，默认简洁版 |
| 未说明粒度 | unknown | 必须触发 human-in-the-loop 二选一选择 |

### Human-in-the-loop 粒度选择

当用户未说明输出粒度，且没有表达“直接生成、快点、先给我看、先出一版”等立即生成意图时，日报、周报、月报都必须先触发 human-in-the-loop 选择，不要用普通文本追问替代。

选择问题：

你希望生成哪种版本？

选项必须只有两个：

- 简洁版：适合快速查看、发给老板/管理者。
- 详细版：展开客户、商机、承诺对账和风险动作。

用户选择后再继续查询和生成报表。日报、周报、月报的粒度确认体验必须一致。

如果用户没有回复或要求继续生成，不要卡住流程，按简洁版输出，并在报表结尾、CRM 保存确认前补一句：

如需，我可以继续展开为详细版。

输出粒度只影响报表展开程度，不影响 CRM 查询、风险识别、日志保存确认和 HTML 输出规则。

## 时间范围解析

| 报告类型 | 默认时间范围 |
|----------|--------------|
| 日报 | 今天；如果用户说“昨日/昨天/某一日/具体日期”，使用用户指定自然日 |
| 周报 | 本周；如果用户说“上周/某一周/Wxx/YYYY-WXX/某个日期所在周”，使用用户指定 ISO 自然周 |
| 月报 | 本月；如果用户说“上月/某一月/具体月份”，使用用户指定自然月 |

输出中必须写明实际采用的时间范围；如果无法确定，标注“时间范围待确认”。

具体周期边界和指标时间字段见 `references/report_periods.md`。

日期解析必须产出同一组 `period_start`、`period_end` 和 `journal_time`，并同时用于报表标题、CRM 查询时间过滤、日志保存预览和写入 payload。不要让标题使用一套日期、查询使用另一套日期、写入日志再使用当前日期。

日报、周报、月报都必须先把用户表达解析为唯一的 `period_start`、`period_end`、`journal_time`。支持并必须正确处理：今日、昨日/昨天、某一日、上周、本周、某一周、上月、本月、某一月。

周报日期必须按 ISO 自然周计算：周一为开始，周日为结束。不要把“本周”“上周”或“某个日期所在周”解析为从今天、昨天、指定日期、周二或任意日期开始的滚动 7 天窗口。相对周必须基于当前运行环境的 `current_date/currentDateTime` 计算：

- `current_week_start = current_date 所在 ISO 周的周一`
- `本周 period_start = current_week_start`
- `本周 period_end = current_week_start + 6 天`
- `上周 period_start = current_week_start - 7 天`
- `上周 period_end = current_week_start - 1 天`
- `某个日期所在周 period_start = 指定日期所在 ISO 周的周一`
- `某个日期所在周 period_end = period_start + 6 天`
- 周报日志写入 `journal_time = period_end` 的 `yyyyMMdd`

如果当前日期是周一，用户说“上周”，`period_end` 必须是昨天（上一个周日），不能包含当前周一。回归校验：当 `current_date = 2026-05-25`（周一，Asia/Shanghai）时，`上周` 必须是 `2026-05-18 至 2026-05-24`，`journal_time = 20260524`；绝不能解析为 `2026-05-19 至 2026-05-25`。本行是回归测试样例，不得在其他日期当作当前日期使用。不要使用规则文档里的示例日期、模型训练日期或上一轮对话日期覆盖当前运行环境日期。

日报日志写入 `journal_time` 使用日报对应日期；月报日志写入 `journal_time` 使用该自然月最后一天。三种周期的 `journal_time` 都必须是 `yyyyMMdd`。

## 个人边界

本技能只处理一个销售人员的个人视角。遇到以下请求时不要继续生成个人报表，应提示切换到销售管理者报表技能：

- 团队日报、团队周报、团队月报
- AE 横向对比、排名、绩效分层
- 主管管理看板、团队风险汇总
- 跨 AE 资源协调

## 数据准备

在进入具体日报、周报或月报规则前，先确认是否有以下数据：

1. 销售人员姓名或账号
2. 时间范围
3. 客户、商机、跟进、任务、成交数据
4. 上期 `JournalObj` 日志或 CRM 历史记录（周报、月报优先需要）

如果数据不足，不要编造。继续输出时只在受影响的业务模块内说明缺少什么，以及这些缺口会影响哪些判断。

人员范围、同名消歧和数据覆盖说明见 `R08_person_scope_and_coverage.md`。

## 端到端处理流程

### Step 1 - 解析销售人员

从用户输入中识别销售姓名、账号或“我”。如果用户说“我”，优先使用当前人员上下文；无法获取时先追问。

如果出现多名销售人员或团队范围，停止个人报表生成，提示用户指定单个销售人员或切换团队报表技能。

### Step 2 - 解析周期与时间窗口

根据用户表达确定日报、周报或月报，并输出实际采用的日期范围。必须同步确定 `period_start`、`period_end` 和 `journal_time`，供后续 CRM 查询、报表标题和日志保存复用。

### Step 3 - 解析或确认输出粒度

根据“输出粒度判断”确定简洁版或详细版。用户已明确要求时直接采用；用户未说明且不急于生成时先询问；用户要求快速生成时默认简洁版。

### Step 4 - 获取数据

按周期读取：

- 个人日报：CRM 当日数据 + `JournalObj` 最近个人日志
- 个人周报：CRM 本周数据 + `JournalObj` 上周日志/本周已生成日志
- 个人月报：CRM 本月数据 + `JournalObj` 上月日志/月内日志

如 CRM 或 `JournalObj` 不可用，基于用户输入输出可判断部分；本地 `memory/` 仅在存在且可信时作为兜底参考。

### Step 5 - 对账承诺

调用 `R05_todo_reconciliation.md`，检查上期承诺、本期待办和完成证据。

### Step 6 - 识别风险

调用 `R06_personal_risk_detection.md`，识别停滞、竞品、Champion、承诺泡沫和 Pipeline 虚高。

### Step 7 - 生成行动

调用 `R07_next_action.md`，输出下一周期 P0/P1/P2。

### Step 8 - 输出报表

根据周期调用：

- `R02_daily_report.md`
- `R03_weekly_report.md`
- `R04_monthly_report.md`

输出必须包含摘要、关键指标、关键成果、客户/商机进展、待办对账、风险预警、下一步行动、数据说明和 CRM 日志保存确认。

数据说明放在报表最后，只展示人员/部门和 CRM 数据覆盖；CRM 数据使用中文对象名，不展示对象 ApiName、记录 ID 或长串主键；正文必须使用引用块 `>` 弱化展示；不默认展示数据缺口与口径。

如果用户要求 HTML 输出，在标准报表内容确认后执行 `R10_html_report_output.md`，生成自适应单页 HTML。HTML 输出后仍需执行 CRM 日志保存确认，未确认前不得写入。
