# R02 - 个人日报

## 目标

个人日报用于回答：今天做了什么、哪些事情有结果、哪些客户有风险、明天先做什么。

日报不是工作流水账。重点写结果、结论、风险和明日动作。

## 输入要求

优先读取或要求用户提供：

- 当日跟进记录：客户、联系人、沟通方式、沟通结论
- 当日任务：计划完成、实际完成、延期、取消
- 当日商机变化：阶段变化、金额变化、赢率变化、预计签约时间变化
- 当日成交或回款
- 明日计划

## 输出结构

普通日报必须直接渲染为正文，不要放进 `markdown`、`text` 或其他代码块。默认模块顺序：

1. `# [销售姓名] 个人日报 - [日期]`
2. `## 🧭 摘要`：今日最重要结果、今日最大风险、明日最重要动作。
3. `## ✅ 今日关键成果`：每条写成“成果一句话 + 证据 + 价值”。
4. `## 🤝 今日跟进记录`：同一客户或商机先合并，再写跟进动作、客户反馈/结论和下一步。
5. `## 📌 今日待办完成情况`：区分已完成、未完成/延期。
6. `## ⚠️ 今日风险`：写客户/商机、风险判断、依据和建议。
7. `## 🎯 明日行动`：分为 `🔴 P0 必须完成`、`🟡 P1 应该推进`、`🔵 P2 有资源再做`。
8. `## 🧾 数据说明`：人员/部门和 CRM 数据覆盖；使用中文对象名，不展示对象 ApiName、记录 ID 或长串主键；正文必须使用引用块 `>` 弱化展示。
9. `## 💾 CRM 日志保存确认`：按 `R09_journal_storage_confirmation.md` 用中文询问是否保存到 CRM 日志。

## 判定规则

- 有明确客户反馈、阶段推进、交付物、成交或回款，才算“关键结果”。
- “联系了、沟通了、发了资料”只算动作，不算结果；必须追问或标注“结论缺失”。
- 当日未完成的 P0 待办，必须进入明日 P0 或说明取消原因。
- 明日 P0 不超过 3 条，避免把所有事情都写成最高优先级。
- 今日关键成果必须写成“结果 + 证据 + 价值判断”，不要把动作堆成流水账。
- 同一个客户的多条记录要合并成一行，按“做了什么、客户反馈是什么、下一步是什么”表达。
- 普通日报直接输出可渲染正文，不要整体包进代码块。
- 除非用户要求表格，今日关键成果、跟进记录、待办和风险默认用短段落或项目符号，不用表格。
- 数据说明放在最后，只展示人员/部门和 CRM 数据覆盖；CRM 数据使用中文对象名，不展示对象 ApiName、记录 ID 或长串主键；正文必须使用引用块 `>` 弱化展示；不默认展示数据缺口表。
- 用户要求 HTML、网页版、手机查看或自适应页面时，先完成日报正文，再执行 `R10_html_report_output.md`，最后执行日志保存确认。
- 输出日报后必须用中文询问是否保存到 CRM 日志；用户确认前不得写入。

## 输出粒度

### 简洁版（默认）

适合快速查看或发给管理者。控制在可快速阅读的长度：

- 摘要 100 字以内。
- 今日关键成果 3-5 条。
- 今日跟进记录只展开重点客户/商机，普通维护客户合并概述。
- 待办完成情况只列已完成关键事项、未完成/延期事项和待确认事项。
- 今日风险只列中高风险或影响明日 P0 的风险。
- 明日行动 P0 不超过 3 条，P1 不超过 3 条，P2 可省略或最多 2 条。

### 详细版

适合复盘过程或检查执行细节：

- 今日跟进记录可展开到 10 个客户/商机。
- 每个重点客户/商机写动作、客户反馈、结论、下一步。
- 待办按事项逐条对账。
- 风险写完整依据、已动作、下一动作和截止时间。
- 可补充普通客户概览，但不要把无结论流水铺满正文。

无论简洁版还是详细版，都必须保留数据说明和 CRM 日志保存确认。

## 日报风险重点

日报优先识别短周期风险：

- 关键客户未回复
- 今日承诺没有交付物
- 会议有动作但无结论
- 客户新增竞品信号
- 关键联系人态度变冷
- 明日 P0 依赖今天未完成事项

## 禁止行为

- 不把所有跟进记录平铺成流水账。
- 不把无结论沟通写成有效推进。
- 不把明日计划写成泛泛的“继续跟进”。
- 不输出团队指标、团队排名或主管管理动作。
