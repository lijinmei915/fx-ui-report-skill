# R08 - 人员范围与数据覆盖

## 目标

确认本次个人报表对应哪一位销售人员，并明确 CRM 事实数据和 CRM 日志数据的覆盖情况。

本规则在 `R01_period_router.md` 之后、CRM 查询和报表生成之前执行。

## 人员解析优先级

按以下顺序确定销售人员：

1. 用户显式指定的销售姓名、账号或员工 ID。
2. 用户使用“我、我的客户、我负责的商机”时，使用当前人员上下文。
3. 用户粘贴的报表或跟进记录中明确出现的销售人员。
4. 用户补充说明。

如果人员无法确定，先追问；不要默认使用任意销售人员。

## 必须消歧的情况

- 同名销售人员。
- 用户只说“我”，但当前人员上下文不可用。
- 用户同时粘贴多个销售人员的数据。
- CRM 查询到的人员与 JournalObj 历史日志中的人员不一致。
- 用户请求看某个团队、多名 AE 或排名。

遇到多名销售人员时，不要继续生成个人报表；提示用户指定单个销售人员，或切换到销售管理者团队报表技能。

## 数据说明输出

每份个人日报、周报、月报都应在最后包含简短数据说明，不使用表格。数据说明是来源备注，不是业务结论，视觉上要弱化。

数据说明直接作为报表正文模块输出，不要放进代码块：

- `## 🧾 数据说明`
> 人员/部门：[销售人员姓名/员工ID，如可得则写部门]  
> CRM 数据：[已查询的中文对象名和数量/状态；如不可用则说明不可用]

普通 Markdown 输出时，数据说明正文必须使用引用块 `>`，呈现为灰色小字或备注效果。不要把数据说明做成醒目的正文段落、表格或卡片。

默认不展示用户输入、history memory、目标数据，也不单独展示数据缺口与口径。
如果缺失影响核心结论，在对应业务模块内用一句话说明，例如“因目标数据缺失，本次不计算目标完成率”。

CRM 数据说明必须面向业务用户表达，不展示对象 ApiName 或数据 ID。

推荐写法：

> CRM 数据：日志 1 条、销售记录 8 条、日程 0 条、商机 50 条、订单 0 条、回款 0 条。

禁止写法：

- 用对象 ApiName、记录 ID 或长串主键描述日志、销售记录、商机、订单、回款。

如果需要说明某条记录，优先使用日志标题、客户名、商机名、时间和业务类型，不用 `_id` 或记录 ID。

## 数据来源标记

关键判断必须标注来源：

| 来源 | 使用方式 |
|------|----------|
| CRM | 作为事实主来源 |
| JournalObj | 作为历史日报、周报、月报和上期承诺主来源 |
| 用户输入 | 作为当前请求直接输入，需区分自述和证据 |
| memory | 仅在 CRM/JournalObj 不可用时作为本地兜底参考 |
| 推断 | 只能用于判断，不写成事实，必须标注待确认 |

## 覆盖处理

- CRM 为空：不要写成“本期无跟进/无成交”，应写“CRM 未查询到或不可用”。
- 用户自述无证据：可以记录，但完成状态标注“待确认”。
- JournalObj 不可用：说明无法读取历史日志，跨期对账可能不完整。
- memory 缺失：不作为数据缺口，不影响报表生成。
- 目标数据缺失：不计算目标完成率，只输出成交、回款、Pipeline 等事实指标。
- 任务对象缺失：不要声称已查询 CRM 任务；可从日程、销售记录、JournalObj 或用户输入推断，并标注来源。

## 生成前检查

进入具体日报、周报或月报前确认：

1. 销售人员已确认，或已标注待确认。
2. 报告周期和时间范围已确认。
3. CRM 事实数据和 JournalObj 历史日志的覆盖状态已说明。
4. 缺失数据不会被当作 0 或正常。
5. 多人数据不会混成一份个人报表。
