# R03 - 个人周报

## 目标

个人周报用于回答：上周承诺的事本周完成了吗、本周哪些客户和商机有推进、哪里卡住、下周先做什么。

周报的核心是承诺兑现和商机推进，不是活动数量堆叠。销冠级周报应像战报：三秒看数字，十秒看风险，三十秒知道下周要干什么。

## 执行顺序

生成个人周报时按以下顺序执行：

1. 读取或整理上周承诺和本周实际数据。
2. 执行 `R05_todo_reconciliation.md` 做待办对账。
3. 汇总本周 KPI、Pipeline、成交、重点客户变化；摘要和关键指标必须优先呈现可确认数字。
4. 对重点客户/商机做销冠级复盘：金额、阶段变化、关键人、动作、承诺、卡点。
5. 执行 `R06_personal_risk_detection.md` 识别风险和假推进，风险必须绑定处理动作和截止时间。
6. 执行 `R07_next_action.md` 生成下周 P0/P1/P2 行动；P0 必须是影响成交、阶段推进或关键承诺兑现的结果型动作。
7. 用户要求 HTML、网页版、手机查看或自适应页面时，执行 `R10_html_report_output.md`。
8. 最后执行 `R09_journal_storage_confirmation.md` 询问是否保存到 CRM 日志。

## 输入要求

优先读取或要求用户提供：

- 上周计划、承诺或待办
- 本周任务完成记录
- 本周客户跟进记录
- 商机阶段、金额、赢率、预计签约时间变化
- 成交、回款、新增客户、新增商机
- Pipeline 期初、期末和变化原因
- 重点客户/商机的关键人、决策链、预算/采购流程、客户承诺、卡点
- 本周有效动作和无效动作，用于沉淀下周打法

## 输出结构

普通周报必须直接渲染为正文，不要放进 `markdown`、`text` 或其他代码块。默认模块顺序：

1. `# [销售姓名] 个人周报 - [周次/日期范围]`
2. `## 🧭 摘要`：本周最重要结果、本周最大风险、下周最重要动作；必须优先写可确认数字。
3. `## 📊 本周关键指标`：可用表格展示指标、本周、对比上周和说明。
4. `## ✅ 本周关键成果`：每条写成“成果一句话 + 证据 + 价值”。
5. `## 📌 待办对账`：区分已兑现、未兑现/延期、待确认。
6. `## 🤝 重点客户与商机进展`：重点客户/商机按六要素输出，同一客户或商机先合并，再写本周进展、当前判断和下周动作。
7. `## 🧠 本周复盘`：写本周有效打法、无效动作和下周调整；没有足够数据时用一句话说明待补证据。
8. `## ⚠️ 风险预警`：写客户/商机、风险等级、风险判断、依据、已采取动作、下一动作和截止时间。
9. `## 🎯 下周行动`：分为 `🔴 P0 必须完成`、`🟡 P1 应该推进`、`🔵 P2 有资源再做`。
10. `## 🧾 数据说明`：人员/部门和 CRM 数据覆盖；使用中文对象名，不展示对象 ApiName、记录 ID 或长串主键；正文必须使用引用块 `>` 弱化展示。
11. `## 💾 CRM 日志保存确认`：按 `R09_journal_storage_confirmation.md` 用中文询问是否保存到 CRM 日志。

## 销冠级周报标准

### 数字先行

- 摘要至少包含 3 类可确认数字：成交/回款、Pipeline、新增商机或客户、跟进客户、有效会议/Demo/POC、待办兑现率。
- 关键指标尽量使用数字；没有数据时写“待确认：[缺哪个数字]”，不要编造。
- 不用“进展不错、整体平稳、待确认较多”替代关键数字。

## 输出粒度

### 简洁版（默认）

适合快速查看、发给老板或管理者。输出必须短而硬：

- 摘要 150 字以内，必须有关键数字、最大风险、下周最重要动作。
- 关键指标保留 5-8 个核心指标。
- 关键成果 3-5 条，只写有证据、有价值的结果。
- 待办对账只列关键兑现、未兑现/延期、待确认事项。
- 重点客户/商机最多展开 3-5 个，普通维护客户不展开。
- 本周复盘只写有效打法、无效动作、下周调整各 1-2 条。
- 风险只列中高风险和影响 P0 的风险。
- 下周行动 P0 不超过 3 条，P1 不超过 3 条，P2 最多 2 条。

### 详细版

适合复盘过程、检查执行细节或给销售本人做深度改进：

- 关键指标可补充口径解释和变化原因。
- 待办对账逐条展开，包含原承诺、新状态、原因、新截止时间、补救动作和验证标准。
- 重点客户/商机最多展开 10 个，每个重点项尽量写齐六要素。
- 客户与商机进展可增加普通客户概览，但普通客户仍应合并，不铺流水。
- 风险写完整依据、已采取动作、下一动作、对象和截止时间。
- 本周复盘可以展开有效打法、无效动作、下周调整，并说明哪些动作要减少或停止。

无论简洁版还是详细版，都必须保留数据说明和 CRM 日志保存确认。详细版不得把所有无结论沟通堆成流水账。

### 承诺闭环

- 每周必须对账上周承诺和本周兑现。
- 每条未兑现承诺必须写清：原承诺、新状态、未兑现原因、新截止时间、补救动作。
- 没有新截止时间的延期，不能算合理延期；连续延期进入风险预警。

### 重点客户/商机六要素

重点客户/商机默认最多展开 5 个。每个重点项尽量写齐：

- 金额：商机金额、预计回款或影响金额；没有则写待确认。
- 阶段：本周阶段变化，或说明阶段未变化。
- 关键人：Champion、EB、采购、业务负责人、技术负责人等。
- 动作：本周做了什么，必须有客户反馈或内部结果。
- 承诺：客户或销售本周形成的明确承诺。
- 卡点：阻碍成交或推进的具体问题。

普通维护客户一行带过，不强制六要素。

### 销售线和产品线分开

如果报表同时包含产品/方案/交付信息，客户进展中要区分：

- 销售推进状态：金额、阶段、关键人、动作、承诺、卡点。
- 产品/方案依赖：PRD、方案、POC、技术验证、客户需求确认等。

不要把产品需求状态误写成销售推进结果。

### 本周复盘

本周复盘必须沉淀方法，而不是重复摘要：

- 有效打法：哪个动作推动了客户反馈、阶段变化或关键承诺。
- 无效动作：哪些动作只是过程，没有带来结论或推进。
- 下周调整：少做什么、多做什么、优先攻哪类客户或哪条关键链路。

## KPI 与 Pipeline 归因

当 Pipeline、成交金额、目标完成率或商机阶段发生明显变化时，必须说明原因。

Pipeline 变化 = 新增商机 + 阶段升阶 + 金额调整 - 成交转出 - 丢单/失效

如果数字对不上，直接标注：

数据不一致：Pipeline 变化缺少解释，需补充变化原因。

不要替销售人员补理由。

## 客户与商机排序

默认按以下优先级排序：

1. 本周成交或接近成交的商机
2. 高金额或关键客户商机
3. 高风险、停滞、竞品相关商机
4. 本周形成明确客户承诺或关键人反馈的商机
5. 本周新增但需要验证意向的商机
6. 普通维护客户

## 周报注意事项

- 摘要不超过 150 字，使用结论先行的金字塔结构。
- 摘要必须有数字；缺数字时明确写“待确认”并说明缺哪个指标。
- 正常推进一行带过，风险和卡点展开说明。
- 只写个人负责范围，不做团队汇总。
- 没有证据的完成标注“待确认”。
- 同一个客户或商机的多条跟进记录要合并，重点项按金额、阶段变化、关键人、动作、承诺、卡点输出。
- P0 只放会影响成交、阶段推进、关键客户承诺或重大风险止损的动作；同步数据、整理记录、普通资料准备不能单独列为 P0。
- 普通周报直接输出可渲染正文，不要整体包进代码块。
- 除关键指标外，关键成果、待办对账、客户进展和风险默认用短段落或项目符号，不用表格。
- 缺少上周承诺或 JournalObj 上期日志时，在待办对账里用一句话说明，不单独展示数据缺口表。
- 数据说明放在最后，只展示人员/部门和 CRM 数据覆盖；CRM 数据使用中文对象名，不展示对象 ApiName、记录 ID 或长串主键；正文必须使用引用块 `>` 弱化展示；不默认展示数据缺口表。
- 用户要求 HTML、网页版、手机查看或自适应页面时，先完成周报正文，再执行 `R10_html_report_output.md`，最后执行日志保存确认。
- 输出周报后必须用中文询问是否保存到 CRM 日志；用户确认前不得写入。
