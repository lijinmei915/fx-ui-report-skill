# R05 - 待办与承诺对账

## 目标

对账用户在上期承诺的事项和本期实际完成情况，判断完成、延期、逾期、取消或待确认。

日报可对账“今日计划 vs 今日完成”；周报对账“上周承诺 vs 本周实际”；月报对账“月初/月内重点承诺 vs 月末结果”。

## 输入解析

按以下优先级提取：

1. CRM 任务对象或明确任务字段。
2. CRM 日程 `ScheduleObj`。
3. CRM 销售记录 `ActiveRecordObj` 中可识别的承诺事项。
4. `JournalObj` 上期个人日志中的下期工作计划。
5. 用户输入。
6. 本地 `memory/`，仅在 CRM/JournalObj 不可用时兜底参考。

提取字段包括：

- 客户或商机名称
- 待办内容
- 原计划完成时间
- 当前状态
- 完成证据或未完成原因
- 新截止时间（如延期）
- 补救动作和验证标准（未完成、延期、逾期时必填）

如果没有上期承诺，先尝试从 `JournalObj` 历史日志读取；仍缺失时标注“未找到上期日志或承诺，无法完整对账”。不要因为本地 memory 缺失而判定数据缺口。

## 状态判定

| 状态 | 判定标准 |
|------|----------|
| completed | 有交付物、客户确认、CRM 状态变化、成交或进入下一阶段 |
| overdue | 截止时间已过，未完成，且没有合理延期说明 |
| renewed | 未完成，但有明确原因和新截止时间 |
| cancelled | 任务不再需要做，且有明确取消原因 |
| unconfirmed | 销售人员说完成，但缺少证据 |

## 证据分级

| 证据 | 置信度 | 处理 |
|------|--------|------|
| 合同、邮件、截图、CRM 状态变更、客户确认 | 高 | 可判定完成 |
| 通话纪要且有明确结论 | 中高 | 可判定完成或阶段推进 |
| AE 自述“已完成”但无证据 | 中 | 标注待确认 |
| “做了、沟通过、推进中、差不多” | 低 | 不算完成，追问结论 |
| 无说明 | 无 | 不算完成 |

## 输出格式

待办对账直接作为报表正文模块输出，不要放进代码块。结构如下：

- `## 待办对账`
- 已完成：[客户/事项] 原承诺为 [承诺内容]，本期已完成。证据：[CRM状态/客户确认/交付物]。
- 延期/逾期：[客户/事项] 原计划 [时间] 完成，目前状态为 [延期/逾期]。原因：[原因]。新动作：[对象、动作、截止时间、验证标准]。
- 待确认：[客户/事项] 销售自述已完成，但缺少 [证据]，需要补充确认。

状态只使用：已完成、逾期、延期、取消、待确认。默认用项目符号输出；只有用户要求明细表时才使用表格。

## 特殊情况

- 上期承诺与本期客户变化冲突：不修改历史承诺，说明“客户情况变化导致取消/延期”。
- 前置依赖未完成：标注被阻塞事项，并写清楚影响链条。
- 连续两期延期：进入风险检测，判断是否为承诺泡沫或客户真实意向不足。
- 同一事项多次改期：必须要求新的验证标准，不能只写“继续跟进”。
- 周报中没有上周承诺原文时，必须写“未找到上周承诺，无法完整对账”，不能把本周动作倒推成上周承诺。
- 未兑现承诺如果没有新截止时间和补救动作，必须进入下周 P0/P1 或风险预警，不能只停留在待办对账。

## 禁止行为

- 不接受“做了”作为完成证据。
- 不隐藏逾期事项。
- 不用模糊话术替代状态判断。
- 不把没有新截止时间的延期判定为合理延期。
- 不把“待确认”作为逃避对账的默认状态；能根据证据判断的必须判断。
