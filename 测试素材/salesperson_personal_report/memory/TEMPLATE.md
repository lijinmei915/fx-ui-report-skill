# 个人销售报表本地兜底 Memory 模板

本模板仅用于 CRM 或 `JournalObj` 不可用时的本地兜底归档。真实运行优先使用 CRM 标准对象、自定义对象和 `JournalObj`，不要把本地 memory 作为默认事实源或归档源。

Memory 只用于单个销售人员视角，不存团队汇总、AE 排名或主管管理看板。

## 记忆架构

```text
memory/
├── TEMPLATE.md
├── daily/
│   └── YYYY-MM-DD/
│       └── [销售姓名].md
├── weekly/
│   └── YYYY-WXX/
│       └── [销售姓名].md
├── monthly/
│   └── YYYY-MM/
│       └── [销售姓名].md
├── todo_history/
│   └── [销售姓名].md
├── customer_context/
│   └── [客户名称].md
└── opportunity_risk/
    └── [商机或客户名称].md
```

## 更新规则

默认不自动更新本地 memory。只有在 CRM/JournalObj 不可用、用户明确要求本地归档，或测试场景需要时才参考以下模板。

| 触发事件 | 建议更新位置 |
|----------|--------------|
| 生成个人日报 | `daily/`，必要时更新 `customer_context/` 和 `opportunity_risk/` |
| 生成个人周报 | `weekly/`、`todo_history/`，必要时更新客户和商机风险 |
| 生成个人月报 | `monthly/`，汇总客户、商机、风险和待办趋势 |
| 客户关键状态变化 | `customer_context/` |
| 商机出现停滞、竞品、Champion 失联 | `opportunity_risk/` |
| 待办完成、延期、逾期、取消 | `todo_history/` |

## 冲突处理

- CRM 与自述冲突：保留两者，标注“待核实”，优先采用 CRM 事实。
- CRM/JournalObj 与历史 memory 冲突：优先采用 CRM/JournalObj，记录冲突点和来源。
- 缺少证据：标注“待确认”，不要写成已完成或确认风险。
- 未获得本地写入权限：只在输出中说明“可选本地归档内容”，不要声称已经写入。
