# 客户360档案

## 目标

为 AE 生成一份用于桌面端深度阅读的客户360档案。这个场景没有“今天必须做什么”的压力，目标是让 AE 建立完整客户认知：

- 这个客户是谁，业务和组织处在什么状态？
- 我们和客户过去怎么打交道，关系如何变化？
- 决策链里谁重要，谁支持，谁阻碍？
- 客户组织里谁影响谁，谁是关键人力地图节点？
- 当前项目为什么启动，客户期望、硬门槛、预算审批和时间表是什么？
- 客户现有 IT 系统、使用规模和优先覆盖群体是什么？
- 当前和历史商机是什么状态？
- 客户真实痛点、竞品格局和未知问题分别是什么？

默认输出结构化 Markdown；当用户明确要求 HTML、网页、可视化简报或清爽版排版时，使用本技能内置 HTML 模板生成标准版客户简报。

## 适用场景

使用本技能处理：

- `/intake [客户名称]`
- `/intake [客户名称] --full`
- “帮我了解这个客户”
- “生成客户360”
- “客户档案 / 客户全貌 / 客户画像”
- “生成 HTML 客户简报 / 网页版客户简报 / 清爽版标准简报”
- “我想先认识这个客户，不是准备马上见客户”

不要用于：

- 见客前 1 分钟行动简报。
- 团队交接或 AE 工作移交。
- 高层陪访、领导汇报或升级汇报。
- 团队周报、客户组合盘点、销售经理 review。
- 客户主体识别、行业分类或产业链位置判断；交给 `company-identifier`。
- 深度决策链诊断；交给 `decision-chain`。
- 深度需求穿透；交给 `needs-discovery`。
- 单商机竞品攻防；交给 `competition-intel`。
- 完整管线预测、目标达成判断或销售预测；交给 `pipeline-review`。

本技能负责生成客户360/客户档案产物，可以展示决策链、痛点、竞品和商机全貌，但不做深度业务诊断。用户要客户主体识别转 `company-identifier`，要决策链诊断转 `decision-chain`，要需求诊断转 `needs-discovery`，要竞品攻防转 `competition-intel`，要管线预测转 `pipeline-review`。

## 输出原则

- 结论先行：正文前必须先输出“结论摘要”，让 AE 先知道客户当前状态和最重要判断。
- 桌面端深度阅读，允许比见客前简报更完整。
- 以事实、关系和历史脉络为主，不强制输出今日行动。
- 所有推断必须标注置信度。
- 不把行业通用痛点当成客户事实。
- 不回避未知；未知项要进入“开放问题”。
- 重点呈现最重要的 10 个关键人（KP），其余人员用简表或归入“其他联系人”。
- 使用少量 emoji 强化章节扫读，只放在章节标题和关键提示，不给每行都加装饰。
- 正文不逐条堆叠“来源/依据”；证据类型和数据口径统一收口到文末“参考来源”弱引用区。
- 不删减已识别内容；数据不足时保留章节并写“未知/待补充”，不要直接省略。
- 标准版可以给出“后续建议动作”，但必须服务客户认知和机会判断，不写成见客前强行动板。

## 数据优先级

1. Memory：历史博弈、关系变化、承诺兑现、关键事件。
2. CRM Account：客户基础信息。
3. CRM Opportunity：当前和历史商机。
4. CRM Contacts：完整决策链和关键人。
5. CRM Activity：最近 6-12 个月互动记录。
6. 合同、订单、回款、CSM 预警、流失原因：full 模式或有历史客户时使用。
7. References：行业基准、竞品参考、CRM 字段说明。
8. 公开信息：官网、新闻、年报，只在用户要求或信息缺口明显时使用。

## 处理流程

1. 识别客户。
   - 需要从 CRM 查询时，先读取 `references/cli_reference.md`。
   - 查询 `AccountObj`，拿到客户 `_id`。
   - 多个候选客户时，先让用户选择。

2. 拉取客户360数据。
   - 默认读取客户、商机、联系人、销售记录、竞品明细。
   - `--full` 或历史客户场景，再读取合同、订单、回款、CSM 预警、流失原因。
   - 如字段含义不清，读取 `references/crm_schema.md`。

3. 读取 Memory。
   - 读取 `memory/customers/[客户名称]/` 下已有记忆。
   - 只有创建或更新记忆结构时，读取 `memory/template.md`。

4. 加载规则。
   - 标准模式：执行 `rules/R01_customer_profile.md` 至 `rules/R08_open_questions.md`。
   - full 模式：额外执行 `rules/R09_history_financial.md`。

5. 生成默认 Markdown 档案。
   - 先输出结论摘要，再展开 1-8 节客户360正文。
   - 输出客户画像、博弈时间线、关系温度、决策链与人力地图、项目背景与商机全貌、痛点地图、竞品全貌、开放问题。
   - 如存在项目型商机，补充项目起源、客户期望、必选条件、预算审批、IT 现状和使用规模。
   - 在开放问题后补充风险矩阵和后续建议动作，建议动作不超过 5 条。
   - `--full` 模式追加历史合作、财务与客户健康。
   - 最后输出弱化的“参考来源”，不要把来源说明散落在正文每条判断后。
   - 不输出“今日目标”“行动 P0/P1/P2”“确认出发”等见客前语言；P0/P1/P2 仅可用于使用规模优先级。

6. 用户要求 HTML 输出时。
   - 先完成 Markdown 结构化内容，再读取 `rules/R10_html_output.md`。
   - 使用 `assets/templates/standard_briefing.html` 的版式和 CSS 风格生成 HTML。
   - HTML 只做展示层，不改变客户判断、置信度和数据口径。

## 资源路由

- 需要从 CRM 拉取数据时，读取 `references/cli_reference.md`。
- 需要解释 CRM 对象或字段时，读取 `references/crm_schema.md`。
- 需要行业规模、采购周期、财务阈值或常见痛点假设时，读取 `references/industry_benchmarks.md`。
- 已知或疑似竞品时，读取 `references/competitive_reference.md`。
- 用户要求 HTML、网页、清爽版排版或可视化简报时，读取 `rules/R10_html_output.md`，并使用 `assets/templates/standard_briefing.html`。
- 只有 full 模式或用户要求历史订单/回款/增购时，读取 `rules/R09_history_financial.md`。
- 本技能的 HTML 模板只服务标准版客户简报；如需见客前、团队交接或高层陪访模板，应切换对应 skill。

## 置信度规范

- `确认`：有 CRM 记录、客户原话、合同/订单/回款记录或明确 memory 事件。
- `推断`：有行为证据支撑，逻辑链清楚，但客户未直接确认。
- `未知`：没有证据，不允许补成事实。

## 禁止行为

- 不输出见客前简报式的“今日目标”“行动 P0/P1/P2”“底线”“确认出发”。
- 不强制给见客前行动板；如有建议，只写服务客户认知、项目推进或风险验证的后续建议动作。
- 不把 CRM 数据简单罗列成字段清单，必须整合成客户认知。
- 不把行业假设、竞品猜测或 AE 感觉写成事实。
- 不输出超过 20 个关键人；重点说明最重要的 10 个关键人（KP）。
