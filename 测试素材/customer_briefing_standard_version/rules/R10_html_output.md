# R10 HTML 输出

## 目标

当用户要求 HTML、网页、可视化简报或清爽版排版时，把标准版客户简报渲染为单文件 HTML。HTML 只负责展示层，客户判断仍以 R01-R09 的结构化内容为准。

## 触发条件

仅在用户明确提出以下诉求时执行：

- “输出 HTML”
- “做成网页”
- “清爽版排版”
- “可视化客户简报”
- “按标准版 HTML 模板输出”

未触发时，默认输出 Markdown。

## 模板资产

使用：

```text
assets/templates/standard_briefing.html
```

该模板是标准客户360简报样式，包含内联 CSS 和轻量主题切换脚本，适合直接保存为 `.html` 文件并上传或预览。

## 输出结构

HTML 必须按以下顺序组织：

1. 顶部头部：客户名称、简报状态、行业/业务摘要、负责人、客户健康度。
2. 结论摘要：一句话判断、主要机会、主要风险、决策链判断、最大信息缺口。
3. 客户背景：基础信息、业务现状、上系统动机/核心痛点。
4. 人力地图：关键人卡片、角色、态度、价值/风险、标签。
5. 项目背景与目标：项目起源、客户期望、必选条件、预算审批、当前阶段、决策时间表。
6. IT 现状：现有系统、用途、满意度、关键信息、机会点。
7. 使用规模：部门/群体、规模、核心诉求、优先级。
8. 拜访/博弈时间线：最近关键节点，计划中节点可置顶。
9. 竞争格局与风险：竞品表、我方优劣势、风险矩阵。
10. 后续建议动作：最多 5 条，按本周内、下周内、关键节点前/当天等时间粒度分组。
11. 总结：一句话判断、关键胜负手、赢单/续约/增购概率或置信度。
12. 页脚弱引用：数据截止、下次更新、参考来源。

## 版式规则

- 使用 `assets/templates/standard_briefing.html` 的视觉语言：1280px 内的客户360阅读仪表盘、统一页面留白、蓝色头部、卡片化模块、紧凑表格、少量标签色。
- 标准版客户简报默认浅色主题，必须支持手动切换深色/浅色模式；主题切换使用 `html.dark`、CSS 变量和 `localStorage`，不依赖外部资源。
- Web 端使用左右两列阅读布局：左侧承载结论摘要、客户背景、人力地图、项目背景与 IT 现状，右侧承载使用规模、时间线、竞争风险、后续动作与总结。移动端仍按正文 1-10 顺序单列展示。
- 页面必须干净、可扫读，不使用大面积装饰图、营销式 hero、渐变堆叠卡片或无意义插图。
- 一级模块使用编号圆角标识；emoji 只用于标题或关键状态，不给每行加装饰。
- 表格用于结构化信息，卡片用于关键人和风险，时间线用于关键互动历史。
- 高风险使用红色，中风险使用黄色，低风险使用绿色；不要用颜色表达没有证据的判断。
- 正文不逐条堆叠“来源/依据”；来源统一放到底部弱引用。
- 数据不足时保留模块，写“未知/待补充/暂无记录”，不要删除模块。
- HTML 不应依赖外部 CDN、远程字体或图片；必须是可离线打开的单文件。
- 移动端保持可读：桌面双栏在小屏折叠为单列，两栏/三栏业务内容继续折叠为单列，表格必须包裹横向滚动容器并提示可横向滑动，长文本使用 `overflow-wrap` 防止溢出。
- 打印模式必须弱化阴影并强制使用浅色变量，避免深色模式下打印大面积深底。

## 禁止行为

- 不把 HTML 样例中的客户名称、人员、金额、时间或具体业务内容复制到新客户简报中。
- 不为了好看而省略风险、未知项、置信度或信息缺口。
- 不把“后续建议动作”写成见客前“确认出发/今日底线/作战板”。
- 不输出需要用户手动拼接 CSS 或脚本的半成品。
- 不把 HTML 模板内容塞进 Markdown 正文。
