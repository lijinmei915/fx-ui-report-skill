# R01 - 团队报告周期路由

## 目标

先判断用户要的是团队日报、团队周报还是团队月报，再决定读取哪些规则和输出什么粒度。

本规则始终先执行。

## 周期判断

| 用户表达 | 报告类型 | 读取规则 |
|----------|----------|----------|
| 今日团队情况、昨天团队情况、昨日团队情况、某天团队情况、某一日、具体日期、团队日报、今日管理看板 | 团队日报 | R08、R02、R05、R06、R07、R09；用户要求 HTML 时加 R10 |
| 本周、上周、某一周、某个日期所在周、团队周报、Wxx、YYYY-WXX、周会复盘 | 团队周报 | R08、R03、R05、R06、R07、R09；用户要求 HTML 时加 R10 |
| 本月、上月、某一月、具体月份、团队月报、月度经营复盘、月度管理报告 | 团队月报 | R08、R04、R05、R06、R07、R09；用户要求 HTML 时加 R10 |

如果用户没有说明周期，优先追问一句：

你想看团队日报、团队周报还是团队月报？如果不指定，我默认生成团队周报。

如果用户同时提到多个周期，例如“本周和本月”，按用户顺序分别输出，不混成一个报告。

## 输出格式判断

识别用户是否要求 HTML 输出。以下表达表示 `output_format = html`：

- HTML
- html
- 网页版
- 手机查看
- 自适应页面
- H5
- 浏览器打开
- 做成页面

HTML 是输出格式，不是报告周期。用户说“生成 HTML 版团队周报”时：

- 报告周期 = 团队周报
- 输出格式 = HTML
- 仍执行团队周报完整分析流程
- 额外执行 `R10_html_report_output.md`
- 最后仍执行 CRM 日志保存确认

## 输出粒度判断

识别用户希望生成简洁版还是详细版，设置 `output_detail`：

| 用户表达 | output_detail | 处理 |
|----------|---------------|------|
| 简洁版、摘要版、快速看、给老板看、只要重点、三分钟看完、管理层版 | concise | 输出简洁版 |
| 详细版、展开、完整、明细、逐个 AE 展开、逐个客户展开、复盘过程、全部跟进记录 | detailed | 输出详细版 |
| 直接生成、快点、先给我看、先出一版 | concise | 不追问，默认简洁版 |
| 未说明粒度 | unknown | 必须触发 human-in-the-loop 二选一选择 |

### Human-in-the-loop 粒度选择

当用户未说明输出粒度，且没有表达“直接生成、快点、先给我看、先出一版”等立即生成意图时，团队日报、团队周报、团队月报都必须先触发 human-in-the-loop 选择，不要用普通文本追问替代。

选择问题：

你希望生成哪种版本？

选项必须只有两个：

- 简洁版：适合快速查看、发给老板/管理层。
- 详细版：展开 AE 表现、重点客户/商机、跨 AE 风险和管理动作。

用户选择后再继续查询和生成报表。团队日报、团队周报、团队月报的粒度确认体验必须一致。

如果用户没有回复或要求继续生成，不要卡住流程，按简洁版输出，并在报表结尾、CRM 保存确认前补一句：

如需，我可以继续展开为详细版。

输出粒度只影响报表展开程度，不影响 CRM 查询、团队聚合、风险识别、日志保存确认和 HTML 输出规则。

## 范围判断

识别用户要看的团队范围：

- 全部团队
- 指定团队或区域
- 指定行业组
- 指定销售经理下属团队
- 指定 AE 列表

如果范围不明确，优先使用当前管理者上下文；如果无法知道当前管理范围，先追问。

范围、AE 名单和数据覆盖率的详细处理见 `R08_team_scope_and_coverage.md`。所有报表进入数据聚合前，必须先确认或标注团队成员范围。

## 时间范围解析

| 报告类型 | 默认时间范围 |
|----------|--------------|
| 团队日报 | 今天；用户指定昨日/昨天/某一日/具体日期时使用指定自然日 |
| 团队周报 | 本周；用户指定上周/某一周/Wxx/YYYY-WXX/某个日期所在周时使用指定 ISO 自然周 |
| 团队月报 | 本月；用户指定上月/某一月/具体月份时使用指定自然月 |

输出中必须写明实际采用的时间范围；如果无法确定，标注“时间范围待确认”。

具体周期边界和指标时间字段见 `references/report_periods.md`。

日报、周报、月报都必须先把用户表达解析为唯一的 `period_start`、`period_end`、`journal_time`。支持并必须正确处理：今日、昨日/昨天、某一日、上周、本周、某一周、上月、本月、某一月。

团队周报日期必须按 ISO 自然周计算：周一为开始，周日为结束。不要把“本周”“上周”或“某个日期所在周”解析为从今天、昨天、指定日期、周二或任意日期开始的滚动 7 天窗口。相对周必须基于当前运行环境的 `current_date/currentDateTime` 计算：

- `current_week_start = current_date 所在 ISO 周的周一`
- `本周 period_start = current_week_start`
- `本周 period_end = current_week_start + 6 天`
- `上周 period_start = current_week_start - 7 天`
- `上周 period_end = current_week_start - 1 天`
- `某个日期所在周 period_start = 指定日期所在 ISO 周的周一`
- `某个日期所在周 period_end = period_start + 6 天`
- 周报日志写入 `journal_time = period_end` 的 `yyyyMMdd`

如果当前日期是周一，用户说“上周”，`period_end` 必须是昨天（上一个周日），不能包含当前周一。回归校验：当 `current_date = 2026-05-25`（周一，Asia/Shanghai）时，`上周` 必须是 `2026-05-18 至 2026-05-24`，`journal_time = 20260524`；绝不能解析为 `2026-05-19 至 2026-05-25`。本行是回归测试样例，不得在其他日期当作当前日期使用。不要使用规则文档里的示例日期、模型训练日期或上一轮对话日期覆盖当前运行环境日期。

团队日报日期必须按 `Asia/Shanghai` 自然日计算：今天为当前运行环境日期 00:00:00 至 23:59:59，昨天为前一自然日。不要使用规则文档示例日期或模型训练日期替代当前运行环境日期。

团队月报日期必须按自然月计算：本月为当前运行环境日期所在月份的 1 日至当月最后一日，上月为前一自然月的 1 日至最后一日。不要使用规则文档示例日期或模型训练日期替代当前运行环境日期。

日报、周报、月报输出前都必须校验标题、正文开头、CRM 查询时间过滤、日志保存时间来自同一个已解析周期；不一致时先修正日期，不要继续输出。

## 边界

本技能只处理团队管理视角。遇到以下请求时应提示使用个人销售报表技能：

- 单个 AE 的个人日报、周报、月报
- 单个 AE 的待办对账
- 单个客户的个人跟进建议
- 只关心个人下一步行动而非管理动作

## 数据准备

进入具体日报、周报或月报规则前，确认是否有：

1. 团队范围或 AE 列表
2. 时间范围
3. CRM 客户、商机、跟进、成交、回款数据；任务数据如存在则读取
4. CRM 日志对象中的历史团队日志或管理动作
5. CRM/日志不可用时的本地 memory 或用户补充

如果数据不足，不要编造。继续输出时只在受影响的业务模块内说明缺少什么，以及这些缺口会影响哪些判断。

## 端到端处理流程

### Step 1 - 解析团队名称与管理范围

从用户输入中识别：

- 团队名称，例如“华东团队”“ICT 组”
- 区域或行业组，例如“华南”“制造业组”
- 管理者身份，例如“张总下面的团队”
- 明确 AE 列表，例如“张明、李华、王芳”

如果用户只说“我的团队”，优先使用当前管理者上下文；无法获取当前管理者或下属范围时，先追问团队范围。

### Step 2 - 解析报告周期与时间窗口

根据用户表达确定：

- 团队日报：具体日期，默认今天
- 团队周报：具体周次或日期范围，默认本周
- 团队月报：具体月份，默认本月

输出标题中必须写明最终采用的时间窗口。

### Step 3 - 解析或确认输出粒度

根据“输出粒度判断”确定简洁版或详细版。用户已明确要求时直接采用；用户未说明且不急于生成时必须触发 human-in-the-loop 二选一选择；用户要求快速生成时默认简洁版。

### Step 4 - 获取 AE 列表

优先顺序：

1. 用户显式提供的 AE 列表。
2. CRM 或 CLI 查询到的团队成员。
3. 当前管理者上下文中可确认的下属范围。
4. CRM/日志不可用时，使用本地 memory 中该团队/范围内出现的候选 AE。
5. 用户补充说明。

如果 AE 列表不完整，标注“团队成员范围待确认”。

调用 `R08_team_scope_and_coverage.md` 生成范围与数据覆盖说明。缺失 AE 不得从团队结论中静默剔除。

### Step 5 - 查询或读取数据

按周期读取：

- 团队日报：CRM 当日数据 + CRM 日志最近团队日志
- 团队周报：CRM 本周数据 + CRM 日志上周团队日志/本周已生成日志
- 团队月报：CRM 本月数据 + CRM 日志上月团队日志/月内日志

本 skill 的 `memory/` 和个人销售报表 memory 仅在 CRM/日志不可用时作为兜底参考，用于团队历史趋势、AE 健康度、跨 AE 风险和管理动作追踪。

### Step 6 - 汇总与加工数据

调用 `R05_team_aggregation.md`，至少按以下维度加工：

- 团队整体
- AE
- 客户/商机
- Pipeline 阶段
- 时间趋势
- 风险类型

加工结果必须保留数据来源；缺失影响核心结论时，在对应模块内说明。

### Step 7 - 检查管理动作闭环

优先读取上一周期 CRM 日志、任务、日程或销售记录：

- 团队日报读取上一日或最近一条管理动作记录
- 团队周报读取上一周团队日志或管理动作记录
- 团队月报读取上一月团队日志或管理动作记录

CRM/日志不可用时才参考 `memory/management_action/`。

检查 P0/P1 是否完成、是否有验证证据、是否继续影响本期结果。未完成动作必须进入本期“管理动作闭环”、风险或行动计划。

### Step 8 - 识别团队风险

调用 `R06_cross_ae_risk.md`，区分：

- 单个 AE 的个体异常
- 关键客户或商机风险
- 跨 AE 系统性风险
- 数据缺失导致的待确认风险

### Step 9 - 生成管理动作

调用 `R07_management_action.md`，输出主管 P0/P1/P2。

每条 P0/P1 必须包含负责人、涉及 AE/客户、动作、截止时间和验证标准。

### Step 10 - 输出报表

根据周期调用：

- `R02_team_daily_report.md`
- `R03_team_weekly_report.md`
- `R04_team_monthly_report.md`

输出必须包含摘要、团队关键指标、关键结果、AE 表现与异常、重点客户与商机、跨 AE 风险、团队行动计划、数据说明和 CRM 日志保存确认。

数据说明放在报表最后，只展示团队/AE 覆盖和 CRM 数据覆盖；正文必须使用引用块 `>` 弱化展示；CRM 数据使用中文对象名，不展示对象 ApiName、记录 ID 或长串主键；不默认展示数据缺口与口径。

如果用户要求 HTML 输出，在标准报表内容确认后执行 `R10_html_report_output.md`，生成自适应单页 HTML。HTML 输出后仍需执行 CRM 日志保存确认，未确认前不得写入。
