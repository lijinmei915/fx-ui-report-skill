# R08 - 团队范围与数据覆盖

## 目标

把用户输入中的团队、区域、行业组、管理者或 AE 列表转换成可统计的团队范围，并明确本期数据覆盖情况。

本规则在 `R01_period_router.md` 之后、CRM 查询和数据聚合之前执行。

## 范围解析优先级

按以下顺序确定 AE 范围：

1. 用户显式给出的 AE 列表。
2. 用户指定的团队、区域、行业组或销售经理，并且 CRM/CLI 能查询到成员。
3. 当前管理者上下文中可确认的下属范围。
4. CRM/日志不可用时，使用本地 memory 中出现的候选 AE。
5. 用户补充说明。

第 4 类只能作为候选范围，不能直接当成完整团队名单。无法确认完整成员时，标注“团队成员范围待确认”。

## 必须避免的默认值

- 不把“有数据的 AE”默认当成“全部团队成员”。
- 不把本地 memory 中出现过的 AE 默认当成当前团队正式成员。
- 不假设 `PersonnelObj` 中一定存在团队、部门、上级或负责人字段。
- 不在同名人员未消歧时直接合并数据。
- 不把离职、转岗、新加入成员的空数据直接判定为执行异常；先标注成员状态待确认。

## 数据说明输出

每份团队日报、周报、月报都应在最后包含简短数据说明，不使用表格。数据说明正文必须使用引用块 `>` 的弱化视觉格式。

数据说明直接作为报表正文模块输出，不要放进代码块；标题仍用普通二级标题，标题下每一行说明都用 `>` 开头：

- `## 🧾 数据说明`
- `> 团队/AE 覆盖：[团队范围、已确认 AE 数、待确认成员；无法确认总数时写“覆盖范围待确认”]`
- `> CRM 数据：[已查询的中文对象名和数量/状态；如不可用则说明不可用]`

默认不展示用户输入、history memory、目标数据，也不单独展示数据缺口与口径。
如果缺失影响核心结论，在对应业务模块内用一句话说明，例如“因 2 名 AE 缺少本周 Pipeline，本周团队 Pipeline 质量判断可能低估风险”。

CRM 数据说明必须面向业务用户表达，不展示对象 ApiName 或数据 ID。

推荐写法：

> CRM 数据：日志 2 条、销售记录 30 条、日程 4 条、商机 50 条、订单 3 条、回款 2 条。

禁止写法：

- 用对象 ApiName、记录 ID 或长串主键描述日志、销售记录、商机、订单、回款。

如果需要说明某条记录，优先使用日志标题、客户名、商机名、时间和业务类型，不用 `_id` 或记录 ID。

## AE 覆盖处理

- 数据完整：纳入团队指标。
- 部分缺失：纳入可确认指标，缺失指标标注为待确认。
- 完全缺失：不默认为 0，也不默认为正常；在受影响模块内说明影响。
- 身份或归属不确定：进入待确认名单，不参与正式团队结论。

如果正式 AE 总数未知，不输出百分比；使用“覆盖范围待确认”。

## 数据来源标记

报表中的关键判断必须标注来源：

| 来源 | 使用方式 |
|------|----------|
| CRM | 作为事实主来源 |
| CRM 日志 | 作为历史团队日报、周报、月报和上期管理动作主来源 |
| 用户补充 | 用于解释背景，需注明来自用户 |
| memory | 仅在 CRM/日志不可用时作为本地兜底参考 |
| 推断 | 只能用于判断，不写成事实，必须标注待确认 |

## 消歧规则

遇到以下情况先消歧或标注待确认：

- 同名 AE、同名客户、同名团队。
- 用户只说“我的团队”，但当前管理者上下文不可用。
- 用户说“全部团队”，但没有公司级或部门级范围。
- CRM 查询到的成员与本地 memory 中成员不一致。
- 团队名称、区域、行业组在 CRM 字段中没有明确映射。

## 聚合前检查

进入 `R05_team_aggregation.md` 前确认：

1. 报告周期和时间范围已确定。
2. 团队范围或 AE 候选范围已列出。
3. 每个 AE 的 CRM、日志、目标数据覆盖状态已标注。
4. 无法确认的成员不会被静默剔除。
5. 缺失数据对团队结论的影响已在相关模块内说明。
